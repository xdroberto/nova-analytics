# ADR-002: Auth stack — Better Auth + Drizzle + Postgres

**Status:** Accepted (2026-07-07); risk retired in Phase 1 · **Deciders:** Lead + Roberto

## Context

Hard constraint: **no Supabase, no managed auth** — users must live in our own Postgres.
Requirements: email/password credentials, hashed passwords, revocable DB-backed sessions,
protected dashboard routes, and workable middleware behavior on Next.js 16.

## Decision

**Better Auth 1.6.x** with the Drizzle adapter on **Postgres 17**.

- Schema generated by `@better-auth/cli` (`user`, `session`, `account`, `verification`).
- Sessions are DB rows (revocable, IP/UA tracked, 7-day expiry) surfaced via an
  HttpOnly/SameSite=Lax cookie.
- Optimistic cookie check in `src/proxy.ts` (Next 16's middleware) + authoritative
  `auth.api.getSession()` in the dashboard layout.
- Built-in rate limiting on auth routes.

## Risk management (spike & stabilize)

Better Auth is a young library, so it was validated **first** (Phase 1) in a time-boxed
spike with an explicit gate: fundamental failure → fall back to **Auth.js v5 credentials +
JWT strategy** (this document is that fallback's record). The spike passed; the fallback
was never exercised. Key integration findings (full detail in `docs/ai-process/BRAIN.md`):
credential flows don't auto-redirect via `callbackURL`; `drizzle-kit push` needs `--force`
in non-TTY; Next 16 renamed middleware to `proxy.ts`; client password validation must match
the server's 8-char minimum.

## Consequences

- Auth is fully portable: any Postgres host works; no vendor lock-in.
- We own session hygiene: a stale-cookie clearing route (`/api/session/clear`) prevents
  redirect loops after DB resets or secret rotation (found by adversarial review).
- Email verification flows are intentionally out of scope for the trial (documented
  limitation).
